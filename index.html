Abstract

The bill manager application will be targeting IT professionals that work as free-lancers. Some key features are considered, the principal use-cases will be the following: sign-in, sign-up, recover password, edit profile, add a new task, track tasks and work done, add a new category of activity, edit categories, add a new project, edit existing projects, view reports or generate an invoice.

Objectives

  1. To design a web-based application the specifically targets the it professionals needs;
  2. To have an efficient way to track their work split into task, based on specific projects, described by a status and an amount of time;
  3. To have an overview of the work done and the amount of income generated depending on different criteria such as time, client or project;
  4. To set the pricing depending on the activity;
  5. To generate invoices for customers with the possibility to edit the billed tasks included;
  6. Generated invoices will be exported in specific formats CSV, JSON or XML;

General metrics

  • The user interface has to be intuitive, user-friendly and pleasing to the eye;
  • The application has to be cross-browser compatible;
  • The maximum amount of uses for the starters is 1000, and concurrent users 200;
  • The application must be scalable.

Branding

Statement: "The simple way to be paid!"

Whether you’re a contractor or freelancer, one of the most important aspects of running a successful business is invoicing clients on time. Unfortunately, it’s something that sometimes is easier said than done. Our web application, Bill Manager, can offer you a helping hand.

Assumptions

  • The application will be web-based;
  • The application is free of use;
  • Any user will access the application through a web-browser, any web-browser;
  • The user will be able to use the application only if he/she is authetificated;
  • The design of the application will be intuitive

Arhitecture of the application

The Bill Manager application wants to provide its users(IT freelancers) a way to keep track of their work and also keep track of the income generated. For designing the application we need to start from some basic questions:

  • The application must support a variety of different clients including desktop browsers, mobile browsers;
  • There is a team of developers (different locations)working on the application;
  • New team members must quickly become productive so the applications needs to be easy to understand and modify;
  • New technologies must be taken into consideration, because of later changes and compatibility issues;
  • High scalability and availability is required;

    • For our needs, the concept of microservices seems to fit because:

  • Each microservice is relatively small (which means more speed);
  • Each service can be deployed independently of other services - easier to deploy new versions of services frequently
  • Each service can be developed and deployed independently.
  • Improved fault isolation.
  • Eliminates any long-term commitment to a technology stack.

    • The biggest challenge with using this approach is deciding when it makes sense to use it. When developing the first version of an application, you often do not have the problems that this approach solves. Moreover, using an elaborate, distributed architecture will slow down development. This can be a major problem for startups whose biggest challenge is often how to rapidly evolve the business model and accompanying application. Using Y-axis splits might make it much more difficult to iterate rapidly. Later on, however, when the challenge is how to scale and you need to use functional decomposition, the tangled dependencies might make it difficult to decompose the application into a set of services.
